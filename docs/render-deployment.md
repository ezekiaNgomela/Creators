# Render deployment and smoke-test plan

This project can be smoke-tested online with a single Render Blueprint. Render's current service model supports web services, static sites, managed Postgres, and Redis-compatible Render Key Value instances, so the recommended setup is:

- `creators-api`: Go web service for `apps/api`.
- `creators-web`: Vite static site for `apps/web`.
- `creators-mobile-web`: Expo Router static web export for `apps/mobile`.
- `creators-postgres`: managed PostgreSQL database.
- `creators-redis`: managed Render Key Value instance used through the Redis client.

## Deploy from the Blueprint

1. Push this repository to GitHub or GitLab.
2. In Render, create a new Blueprint from the repository root. Render looks for `render.yaml` at the repository root by default.
3. Apply the Blueprint. It provisions the Go API, both static frontends, Postgres, and Key Value.
4. After the first deploy, open the API health endpoint:
   - `https://creators-api.onrender.com/api/health`
5. Open the frontends:
   - Vite app: `https://creators-web.onrender.com`
   - Expo web app: `https://creators-mobile-web.onrender.com`


## What to enter in Render manually

If you do not want to use the Blueprint yet, create the services manually with these build-only settings. These are the values Render asks for in the free Static Site/Web Service forms.

### Vite web static site

- Render service type: `Static Site`
- Root directory: leave empty / repository root
- Build command: `cd apps/web && npm ci && npm run build`
- Publish directory: `apps/web/dist`
- Environment variable: `VITE_API_ORIGIN=<your Render API external URL>` (for example, `https://creators-api.onrender.com`)
- Rewrite rule: `/*` to `/index.html`

### Expo web static site

- Render service type: `Static Site`
- Root directory: leave empty / repository root
- Build command: `cd apps/mobile && npm ci && npx expo export --platform web`
- Publish directory: `apps/mobile/dist`
- Environment variable: `EXPO_PUBLIC_API_ORIGIN=<your Render API external URL>` (for example, `https://creators-api.onrender.com`)
- Rewrite rule: `/*` to `/index.html`

### Go API web service

- Render service type: `Web Service`
- Runtime: `Go`
- Root directory: leave empty / repository root
- Build command: `cd apps/api && go mod download && go build -o /tmp/creators-api .`
- Start command: `/tmp/creators-api`
- Health check path: `/api/health`
- Environment variables:
  - `DATABASE_URL`: your Render Postgres internal connection string
  - `REDIS_URL`: your Render Key Value internal connection string
  - `JWT_SECRET`: a generated secret value
  - `JWT_ISSUER=creators-auth`
  - `DISABLE_MINIO_HEALTH=true`
  - `FRONTEND_ORIGIN=https://creators-web.onrender.com,https://creators-mobile-web.onrender.com`

The repository also includes `render.yaml`, which contains the same values in one file for Render Blueprint deployment.

## Environment variables

The Blueprint wires the backend to Render-managed data services:

- `DATABASE_URL` comes from `creators-postgres`.
- `REDIS_URL` comes from `creators-redis`.
- `JWT_SECRET` is generated by Render.
- `DISABLE_MINIO_HEALTH=true` keeps the smoke-test health check green without requiring a MinIO service.
- `FRONTEND_ORIGIN` allows both hosted frontend origins through CORS.

The frontends are built against the Render API URL. In the Blueprint, Render injects the API service URL from `creators-api` via `RENDER_EXTERNAL_URL`, and the apps append `/api` in code:

- Vite uses `VITE_API_ORIGIN` from the `creators-api` service.
- Expo web uses `EXPO_PUBLIC_API_ORIGIN` from the `creators-api` service.

If you configure services manually, use the API service external URL without the `/api` suffix for these origin variables. The apps also still support `VITE_API_BASE_URL` and `EXPO_PUBLIC_API_BASE_URL` when you need to provide a complete URL that already includes `/api`.

If a static frontend build is missing these variables, hosted Render frontends fall back to the default API service URL, `https://creators-api.onrender.com/api`. Do not rely on Render's generated frontend hostname suffixes to find the API; for example, `https://creators-web-whtz.onrender.com` should still call `https://creators-api.onrender.com/api` unless you explicitly set a different API URL in Render.

If Render assigns a different API hostname or you rename a service, update these values in the Render dashboard or in `render.yaml`:

- API `FRONTEND_ORIGIN`
- Vite `VITE_API_ORIGIN` or `VITE_API_BASE_URL`
- Expo `EXPO_PUBLIC_API_ORIGIN` or `EXPO_PUBLIC_API_BASE_URL`

## Backend hosting recommendation

For testing, the best backend option is the `apps/api` Go service as a Render web service plus managed Render Postgres and Render Key Value. This is simpler than deploying the experimental multi-service backend because the API already exposes the application routes under `/api`, seeds its database on startup, and uses standard `DATABASE_URL` and `REDIS_URL` connection strings.

For production, add object storage before enabling heavy media usage. Render service disks are useful for persistence on a single service, but S3-compatible object storage is a better long-term fit for uploaded media and rendered clips.

## Smoke-test checklist

Run these checks after each deploy:

```bash
curl https://creators-api.onrender.com/api/health
```

Expected result: JSON with `status: "ok"`, `postgres: "up"`, `redis: "up"`, and `minio: "skipped"`.

Then test both frontends in a browser:

1. Open the Vite site and create/login to a test user.
2. Open the Expo web site and create/login to the same test user.
3. Confirm feed, profile, chat, notifications, and live screens load without CORS errors.
